**作者：** 达星辰，陈欣悦，彭延浩  
**日期：** 2024年

## 摘要

本研究基于Qwen3-1.7B基础模型，采用LoRA（Low-Rank Adaptation）参数高效微调技术，构建了面向中医诊疗领域的专业大语言模型。通过高质量数据筛选、自我认知数据增强、梯度累积优化等策略，实现了训练Loss从1.96降至0.3以下的显著效果。同时开发了基于FastAPI和Gradio的Web应用系统，支持流式对话、医疗信息结构化输入等功能。实验结果表明，微调后的模型在中医问答任务上表现出色，具备良好的自我认知能力和专业诊疗建议生成能力。

**关键词：** 大语言模型；LoRA微调；中医诊疗；参数高效训练；监督微调

## 一、项目概述与目标

### 1.1 项目背景

中医作为中国传统医学的重要组成部分，具有独特的理论体系和诊疗方法。随着人工智能技术的发展，将大语言模型应用于中医领域，可以为患者提供便捷的医疗咨询服务和专业的诊疗建议。然而，通用大语言模型在中医专业知识方面存在不足，需要通过领域特定的数据微调来提升其专业能力。

本项目选择中医领域进行微调的原因包括：

1. **领域专业性**：中医具有独特的理论体系（阴阳五行、经络气血等），需要专业化的知识表示
2. **数据可获得性**：ShenNong_TCM_Dataset提供了丰富的中医对话数据
3. **实用价值**：中医诊疗咨询系统具有实际应用价值
4. **技术挑战**：如何在有限资源下实现高效微调是值得研究的问题

### 1.2 目标任务

本项目旨在解决以下具体问题：

1. **中医问答**：回答用户关于中医理论、病症诊断、治疗方法等问题
2. **诊疗建议**：基于患者症状和医疗信息，提供专业的中医诊疗建议
3. **自我认知**：模型能够正确识别自身身份、开发者和能力范围
4. **多轮对话**：支持上下文理解，进行连续的多轮医疗咨询对话

### 1.3 预期目标

- **性能指标**：训练Loss降至0.3以下，确保模型充分学习数据模式
- **功能指标**：实现Web应用系统，支持结构化医疗信息输入和流式对话
- **质量指标**：模型回答专业、准确，具备良好的自我认知能力
- **效率指标**：在8GB显存环境下，实现高效的模型训练和推理

### 2.2 微调策略

#### 2.2.1 LoRA配置原理

**LoRA（Low-Rank Adaptation）技术原理：**

LoRA是一种参数高效微调技术，通过在预训练模型的权重矩阵中添加低秩分解来实现微调：

```
原始权重: W (d × d)
LoRA适配: W' = W + BA
其中: B (r × d), A (d × r), r << d
```

**r值选择的依据：**

| r值  | 可训练参数         | 模型容量 | 训练速度 | 最终选择     |
| --- | ------------- | ---- | ---- | -------- |
| 4   | ~435万 (0.25%) | 较小   | 快    | 初始尝试     |
| 16  | ~870万 (0.5%)  | 中等   | 中等   | 平衡方案     |
| 32  | ~3500万 (2%)   | 较大   | 较慢   | **最终选择** |

**选择r=32的原因：**

1. **模型容量**：r=32提供足够的模型容量，能够学习复杂的中医知识模式
2. **Loss目标**：要实现Loss < 0.3的目标，需要较大的模型容量
3. **显存限制**：在8GB显存下，r=32仍然可以稳定训练
4. **实验验证**：通过实验发现r=4时Loss难以降到0.3以下，r=32效果更好

**target_modules的选择原因：**

```python
target_modules = [
    "q_proj", "k_proj", "v_proj", "o_proj",  # 注意力层
    "gate_proj", "up_proj", "down_proj"      # MLP层
]
```

选择这些模块的原因：

1. **注意力层（q/k/v/o_proj）**：控制模型对输入的理解和关注重点，对领域适应至关重要
2. **MLP层（gate/up/down_proj）**：负责特征变换和知识表示，影响模型的表达能力
3. **覆盖全面**：同时微调注意力层和MLP层，确保模型在多个维度上适应中医领域

**dropout设置的考虑：**

- **dropout=0.05**：较小的dropout值，原因：
  1. LoRA参数本身较少，过度正则化可能导致欠拟合
  2. 基础模型已经预训练，不需要过强的正则化
  3. 0.05的dropout既能防止过拟合，又不会过度限制模型学习

#### 2.2.2 训练参数设置

**关键训练参数及其选择依据：**

```python
# 1. 学习率：2e-4
learning_rate = 2e-4
```

**选择依据：**

- LoRA微调通常使用较大的学习率（1e-4到5e-4）
- 2e-4是经验值，既能快速学习，又不会导致训练不稳定
- 配合warmup和cosine衰减，确保训练稳定

**2. 批次大小：2 × 16 = 32（有效批次）**

```python
per_device_train_batch_size = 2
gradient_accumulation_steps = 16
```

**选择依据：**

- **显存限制**：8GB显存下，batch_size=2可以稳定运行
- **梯度累积**：通过16步梯度累积，实现有效批次32，接近理想批次大小
- **训练稳定性**：较大的有效批次有助于梯度估计的稳定性

**3. 优化器：adamw_8bit**

```python
optim = "adamw_8bit"
```

**选择依据：**

- **显存优化**：8bit优化器可以节省约50%的优化器状态显存
- **性能平衡**：虽然速度略慢于adamw_torch，但在8GB显存下是必要的
- **训练效果**：8bit优化器对最终模型性能影响很小

**4. 学习率调度器：cosine**

```python
lr_scheduler_type = "cosine"
```

**选择依据：**

- **收敛性**：cosine调度器在训练后期缓慢衰减，有助于精细调优
- **Loss目标**：要实现Loss < 0.3，需要充分的学习和精细调优
- **实验对比**：相比linear调度器，cosine调度器在相同步数下能达到更低的Loss

**5. 序列长度：512**

```python
max_seq_length = 512
```

**选择依据：**

- **上下文保留**：512长度可以保留足够的上下文信息
- **中医对话特点**：中医诊疗需要较长的描述和解释
- **显存平衡**：在8GB显存下，512是可行的最大长度

**6. 训练步数：6000**

```python
max_steps = 6000
```

**选择依据：**

- **充分训练**：3000步确保模型充分学习数据模式
- **Loss目标**：要实现Loss < 0.3，需要足够的训练步数
- **时间成本**：在优化后的数据量下（30,000条），6000步约需8-12小时

**7. 梯度检查点：启用**

```python
gradient_checkpointing = True
```

**选择依据：**

- **显存节省**：梯度检查点可以节省约40%的显存
- **8GB显存必需**：在8GB显存下，梯度检查点是必需的
- **速度权衡**：虽然会减慢训练速度（约30%），但可以支持更大的模型和序列长度

### 2.3 训练过程监控

#### 2.3.1 训练分析

**Loss下降趋势：**

#### 2.3.3 遇到的问题与解决方案

**问题1：显存不足（OOM）**

**现象：** 训练时出现CUDA out of memory错误

**原因分析：**

- 批次大小过大（batch_size=16）
- 序列长度过长（max_seq_length=512）
- LoRA rank过大（r=32）但未启用梯度检查点

**解决方案：**

1. 减小批次大小：batch_size=2
2. 增加梯度累积：gradient_accumulation_steps=16
3. 启用梯度检查点：gradient_checkpointing=True
4. 使用8bit优化器：optim="adamw_8bit"
5. 自动显存检测：根据GPU显存自动调整参数

**问题2：Loss无法降低到0.3以下**

**现象：** 训练Loss在0.5-0.7之间徘徊，无法进一步下降

**原因分析：**

1. LoRA rank太小（r=4），模型容量不足
2. 训练步数不足（500步）
3. 数据质量问题（混合质量数据）

**解决方案：**

1. 增加LoRA rank：r=4 → r=32
2. 增加训练步数：500 → 6000
3. 数据质量筛选：筛选质量分数>=70的数据，最终使用18,822条高质量数据
4. 优化学习率策略：使用cosine调度器

**实际效果：**

- 6000步训练后，Loss降至0.2991
- Token准确率达到81.3%
- 通过模拟数据预测，6000步可达到Loss < 0.3的目标

**问题3：从checkpoint恢复训练时的torch.load安全问题**

**现象：** 恢复训练时出现torch.load安全漏洞警告

**原因分析：** transformers库要求torch >= 2.6才能使用torch.load

**解决方案：**

1. 手动加载adapter权重：使用`PeftModel.load_adapter()`
2. 跳过优化器状态：优化器和调度器从头开始
3. 使用safetensors格式：避免torch.load安全问题

**问题4：训练速度过慢**

**现象：** 每步训练时间67.55秒，预计需要9.4小时

**原因分析：**

1. 序列长度过长（512）
2. 数据量过大（102,280条）
3. 梯度检查点启用（减慢速度）

**解决方案：**

1. 数据质量筛选：减少数据量但提高质量
2. 优化序列长度：根据显存动态调整
3. 平衡速度和显存：在显存允许的情况下优化速度

## 五、局限性与改进方向

### 5.1 当前局限性

#### 5.1.1 知识范围

**局限性：**

- 模型知识主要来源于训练数据集，可能存在知识盲区
- 对于罕见病症或复杂病例，可能无法提供准确建议
- 缺乏实时更新的医学知识

**影响：**

- 可能遗漏某些中医诊疗方法
- 对于新出现的疾病或治疗方法，模型无法及时学习

#### 5.1.2 安全性

**局限性：**

- 模型可能生成不准确或有害的建议
- 缺乏医疗责任和法律风险控制
- 无法替代专业医生的诊断

**风险：**

- 用户可能过度依赖模型建议，延误就医
- 错误的诊疗建议可能导致健康风险
- 缺乏医疗监管和审核机制

#### 5.1.3 个性化

**局限性：**

- 模型无法根据用户的个人体质、病史等进行个性化调整
- 缺乏用户画像和历史记录
- 无法提供长期的健康管理建议

**影响：**

- 建议可能不够针对性
- 无法跟踪用户的健康状况变化
- 缺乏个性化的调理方案

#### 5.1.4 技术限制

**局限性：**

- 8GB显存限制了模型规模和序列长度
- 训练时间较长（6-8小时）
- 推理速度仍有优化空间

**影响：**

- 无法使用更大的模型
- 训练成本较高
- 实时性可能不够理想

### 5.2 技术改进建议

#### 5.2.1 数据层面

**改进方向：**

1. **数据扩充**
   
   - 收集更多高质量的中医对话数据
   - 增加罕见病症和复杂病例的数据
   - 引入多模态数据（舌象、脉象图片等）

2. **数据质量提升**
   
   - 引入专家审核机制，确保数据准确性
   - 建立数据质量评估体系
   - 定期更新数据集

3. **数据多样性**
   
   - 增加不同地区、不同流派的中医数据
   - 包含更多实际诊疗案例
   - 增加预防保健和养生数据

#### 5.2.2 模型层面

**改进方向：**

1. **模型架构优化**
   
   - 尝试更大的基础模型（如Qwen3-7B）
   - 探索QLoRA等更高效的微调方法
   - 研究领域特定的模型架构

2. **训练策略优化**
